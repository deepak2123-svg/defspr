2. Role and Scope Model

Access is not decided by role alone. Every user must have an organization scope.

Identity

User login confirms who the person is.

+
Membership

Group, institute, class, or section scope.

+
Permission

What actions are allowed in that scope.

=
Allowed View / Action

No scope means no access.

Role Scope Primary responsibility
Super Admin All Ledgr clients Create groups/institutes, invite first admins, switch scope, manage pricing and credits.
Group Admin One institute group Manage institutes inside the group, invite institute admins, view group reports and credit usage.
Institute Admin One institute Create classes/sections, invite teachers/students, activate students, monitor tests and scores.
Teacher Assigned classes/sections/subjects Plan tests, view assigned student scores, support report sharing where allowed.
Student Own account only Attempt tests and view own results according to institute settings.

3. Invite-Based Onboarding

No open self-join. Users enter the system by invite or approved membership.

Standalone Institute
1. Super Admin Creates institute and billing profile.
2. Invite Institute Admin Admin accepts invite and activates account.
3. Setup Structure Classes, sections, teachers, students.
4. Start Testing Activate students and consume prepaid credits.
Institute Group
1. Super Admin Creates group and imports institutes.
2. Invite Group Admin Group admin sees only their group.
3. Invite Institute Admins Each branch gets its own admin scope.
4. Branch Operations Classes, teachers, students, tests, reports.
Security baseline: no invite or approved membership means no organization access. A teacher from ZEE Delhi cannot see ZEE Pune, another group, or unassigned sections.

4. Super Admin Scope Switching

Fast operations first; heavier impersonation controls can be added later.

MVP

One login, many scopes

  • Super Admin signs in once.
  • Top bar scope switcher: All Ledgr, group, institute, class, section.
  • Visible banner always shows the current scope.
  • Backend still checks scope on every API action.
  • Basic audit logs for create, update, and delete actions.
Later

Support mode / login as user

  • View exactly what a teacher or student sees.
  • Require support reason for sensitive impersonation.
  • Track support session start/end and sensitive actions.
  • Optional notification to group or institute admin.

3. Student uploaded No credit used yet.
